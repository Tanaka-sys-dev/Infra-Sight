# Sprint 3: Telemetry Simulator and Ingestion Pipeline

## Overview
Sprint 3 adds a telemetry simulation and ingestion pipeline for InfraSight. It generates realistic infrastructure telemetry, aggregates readings into observation windows, writes live data to Firestore, updates device health status, and exposes API endpoints for manual simulation, ingestion, polling, and stream retrieval. This prepares the data contract that Sprint 4 will use for Random Forest prediction.

## Architecture
```text
Device -> Simulator -> Per-Device Buffer -> Window Builder -> Firestore telemetry_windows -> Prediction Engine (Sprint 4)
                         |                                      |
                         v                                      v
                  Device Status Updater                  Frontend dashboards
```

## Simulator
The simulator supports five device profiles and seven scenario modes. Each generated reading contains `deviceId`, `deviceType`, `scenarioMode`, UTC `timestamp`, `isAnomaly`, and eight metrics.

### Device Profiles
| Device Type | CPU | Memory | Disk | Packet Loss | Latency | Interface Errors | Restarts | Uptime |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| server | 5-95 normal <=65 | 20-92 normal <=70 | 30-88 normal <=65 | 0-8 normal <=2 | 2-120 normal <=25 | 0-25 normal <=3 | 0-5 normal <=0 | 85-100 normal >=95 |
| switch | 5-85 normal <=45 | 10-80 normal <=55 | 5-60 normal <=40 | 0-15 normal <=3 | 1-80 normal <=15 | 0-50 normal <=5 | 0-4 normal <=0 | 80-100 normal >=95 |
| router | 5-90 normal <=55 | 15-85 normal <=60 | 5-55 normal <=35 | 0-12 normal <=2 | 1-100 normal <=20 | 0-35 normal <=4 | 0-4 normal <=0 | 82-100 normal >=95 |
| access_point | 5-80 normal <=45 | 10-75 normal <=50 | 5-50 normal <=30 | 0-20 normal <=4 | 3-150 normal <=30 | 0-40 normal <=6 | 0-6 normal <=0 | 75-100 normal >=90 |
| workstation | 2-98 normal <=70 | 15-95 normal <=75 | 20-90 normal <=70 | 0-10 normal <=2 | 3-60 normal <=20 | 0-15 normal <=2 | 0-8 normal <=1 | 70-100 normal >=85 |

### Scenario Modes
| Scenario | Effect |
| --- | --- |
| normal | Values stay within normal profile ranges. |
| high_cpu_overload | CPU 80-98 and memory 75-95. |
| packet_loss_surge | Packet loss 8-20 and interface errors 10-40. |
| restart_instability | Restart frequency 3-8 and uptime 70-88. |
| high_latency | Latency 80-150 and packet loss 5-15. |
| memory_pressure | Memory 80-98 and disk utilisation 75-95. |
| recovery | Anomalous values trend 20 percent toward normal each call. |

## Window Builder
The window builder aggregates two or more readings into an observation window. It computes mean, max, and min for all eight metrics, records the window start/end timestamps, and assigns a `normal` or `fault_prone` label.

### Fault Thresholds
| Feature | Threshold |
| --- | --- |
| cpuUsage_mean | fault if > 75 |
| memoryUsage_mean | fault if > 78 |
| packetLoss_mean | fault if > 5 |
| interfaceErrorCount_mean | fault if > 8 |
| restartFrequency_total | fault if > 2 |
| latency_mean | fault if > 60 |
| uptimePattern_mean | fault if < 90 |

## Telemetry Pipeline
1. A reading is generated by the simulator or received via ingestion.
2. The reading is appended to an in-memory per-device buffer.
3. Device health status is computed and upserted into `device_status`.
4. When 10 readings are available, a telemetry window is built.
5. The window is stored in the Firestore `telemetry_windows` collection.
6. The frontend reads streams and device status through the API.

## New API Endpoints

### POST `/api/telemetry/ingest`
Request fields: `deviceId`, `deviceType`, `metrics`, optional `scenarioMode`, optional `source`.

Example request:
```json
{"deviceId":"srv-001","deviceType":"server","metrics":{"cpuUsage":88.4,"memoryUsage":82.1,"diskUtilisation":61.2,"packetLoss":6.3,"latency":74.6,"interfaceErrorCount":12,"restartFrequency":2,"uptimePattern":87.3},"scenarioMode":"high_cpu_overload"}
```

Example response:
```json
{"ingested":true,"deviceId":"srv-001","healthState":"warning","windowBuilt":false,"window":null}
```

### POST `/api/telemetry/simulate`
Request fields: `deviceId`, `deviceType`, optional `scenarioMode`, optional `count`.

Example request:
```json
{"deviceId":"srv-001","deviceType":"server","scenarioMode":"normal","count":5}
```

Example response:
```json
{"deviceId":"srv-001","scenarioMode":"normal","readingsGenerated":5,"readings":[],"windowBuilt":false,"window":null,"healthState":"healthy"}
```

### POST `/api/telemetry/poll`
No request body. Polls all Firestore devices and returns summaries.

Example response:
```json
{"polled":8,"results":[],"timestamp":"2026-05-13T15:00:00+00:00"}
```

### GET `/api/telemetry/stream/{device_id}`
Query fields: optional `limit` with default 20 and max 100.

Example response:
```json
{"deviceId":"srv-001","readings":[],"count":0}
```

## Background Polling
FastAPI starts a daemon background thread on application startup. Every 30 seconds it loads devices from Firestore, generates one reading per device, updates `device_status`, and builds windows when buffers reach 10 readings. Exceptions are logged and do not stop the polling loop.

Manual polling:
```powershell
Invoke-RestMethod "http://127.0.0.1:8000/api/telemetry/poll" -Method POST
```

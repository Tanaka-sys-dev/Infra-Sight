Section 1: Executive Summary

InfraSight is a full-stack ICT infrastructure monitoring and predictive maintenance system developed for Great Zimbabwe University. It enables ICT administrators to monitor institutional network and computing infrastructure, detect abnormal device behaviour, predict possible device faults before they cause outages, generate and manage alerts, validate fault scenarios, and present measurable evaluation evidence.

The system uses a Random Forest machine learning model trained on telemetry data from five device categories including servers, switches, routers, access points, and workstations. It classifies each device as normal or fault-prone based on eight monitored metrics.

Key results demonstrated by the system

  • 81 percent reduction in fault detection time from 96 minutes to 18 minutes
  • 65 percent reduction in response time from 142 minutes to 49 minutes
  • 68 percent reduction in outage duration from 128 minutes to 41 minutes
  • 5.4 percent improvement in system uptime from 92.4 percent to 97.8 percent
  • Random Forest model accuracy of 100 percent on balanced synthetic training data

Section 2: System Purpose and Objectives

2.1 Purpose

InfraSight reduces the time taken to detect and respond to ICT infrastructure faults at Great Zimbabwe University. Traditional fault handling depends on manual observation or user complaints. InfraSight improves this through continuous telemetry collection, machine learning prediction, and automated alert generation.

2.2 Research Objectives

  • Implement real-time ICT infrastructure monitoring
  • Design a telemetry collection and simulation pipeline
  • Train a Random Forest classifier for fault prediction
  • Generate and manage predictive alerts
  • Validate the system through scenario-based testing
  • Demonstrate operational improvement through evaluation

2.3 Target Users

  • ICT Administrators
  • Network Support Staff
  • System Administrators
  • University Technical Support Teams
  • Examiners and Academic Supervisors

Section 4: System Architecture

InfraSight follows a three-tier client-server architecture: presentation tier using React frontend on Vercel, application tier using FastAPI backend on Render, and data tier using Firebase Firestore cloud database.

4.2 Component Diagram

Browser Client | HTTPS | React Frontend (Vercel) | Firebase Auth and REST API | FastAPI Backend (Render) | Firestore (GCP) and ML Model (rf_model.joblib)

4.3 Data Flow Description

  • Device telemetry generated by simulator
  • Telemetry stored via POST /api/telemetry/ingest
  • Window builder aggregates 10 readings
  • Window stored in Firestore telemetry_windows
  • Random Forest model classifies window
  • Prediction stored in Firestore predictions
  • Alert generated if fault_prone above threshold
  • Alert stored in Firestore alerts
  • Frontend reads and displays all data

Section 9: Random Forest Model

The prediction engine classifies devices as normal or fault_prone using telemetry-derived features.

9.1 Model Configuration

  • Algorithm: Random Forest Classifier
  • Library: scikit-learn
  • Number of trees: 200
  • Training split: 70 percent training, 30 percent testing
  • Class weight: balanced
  • Random state: 42
  • Model version: rf-v1
  • Model file: rf_model.joblib

9.2 Training Dataset

  • Total samples: 500
  • Normal samples: 250
  • Fault prone samples: 250
  • Features: 8 telemetry metrics
  • Labels: normal or fault_prone

9.4 Feature Importance Explanation

Feature importance shows which telemetry metrics influence predictions most strongly. Packet loss was identified as the top predictor in the current model. Other important features include interface error count, CPU usage, restart frequency, and latency. Feature importance supports the academic requirement for explainable machine learning.

9.5 Academic Note on Synthetic Data

The model achieves perfect scores on the synthetic training dataset because the simulator generates clearly separable class boundaries by design. In real-world deployment, performance metrics would depend on actual infrastructure telemetry and could differ from these results. This is standard practice for academic prototype systems and is acceptable for dissertation evaluation purposes.

Section 11: User Guide - Getting Started

A modern web browser such as Chrome, Firefox, or Edge is required. Internet connection is required for Firebase access, or locally running backend and frontend can be used for development. Local backend runs at http://localhost:8000 and frontend runs at http://localhost:5173 or http://localhost:3000. Login with admin@gzu.ac.zw and InfraSight@2026!.

Section 12: User Guide - Dashboard

The dashboard is the main command center and provides a real-time overview of all monitored ICT infrastructure. It includes six KPI cards: Total Devices, Online Devices, Active Alerts, Fault Prone Devices, Predictions Today, and System Uptime or Model Accuracy indicator. The device health grid uses green for healthy, amber for warning, and red for critical. Recent Alerts and Recent Predictions panels show the last five items. Quick actions include Run All Simulations, Refresh Dashboard, and View Evaluation. Demo Mode adds a yellow banner, floating guide, and 10-second auto-refresh.

Section 13: User Guide - Device Monitoring

The Devices page lists all monitored infrastructure and allows users to open device detail pages. The detail page shows device name, type, location, IP address, status chip, prediction badge, last seen timestamp, eight KPI metric cards, telemetry charts, simulation action, prediction action, and recent alerts. KPI cards cover CPU usage, memory usage, disk utilisation, packet loss, latency, interface error count, restart frequency, and uptime pattern. Simulate Reading generates a new telemetry reading and Predict Now sends current metrics to the Random Forest model.

Section 14: User Guide - Predictions

The Predictions page shows model version rf-v1, accuracy, precision, recall, F1 score, training sample count, recent predictions, prediction filters, feature importance chart, and confusion matrix. The predictions table shows device ID, predicted class, confidence, top feature, and timestamp. The confusion matrix identifies true negatives, false positives, false negatives, and true positives.

Section 15: User Guide - Alert Management

The Alerts page shows all system alerts, statistics, filters, and action buttons. Alerts can be filtered by status, severity, and search keyword. Each row shows severity, device ID, alert type, message, status, timestamp, and actions. Acknowledge changes an active alert to acknowledged. Resolve changes an acknowledged alert to resolved. Critical alerts require immediate attention, warning alerts require investigation, and info alerts indicate routine or recovery events.

Section 16: User Guide - Scenario Testing

The Scenarios page provides fault scenario validation. Statistics show total runs, pass count, fail count, and average confidence. Available scenarios include High CPU Overload, Packet Loss Surge, Restart Instability, and Normal Steady State. Running a scenario generates simulated readings, builds an observation window, runs prediction, and displays expected class, predicted class, confidence, result, and alert status. The run history table records scenario name, device, expected class, predicted class, result, confidence, and timestamp.

Section 17: User Guide - Evaluation Dashboard

The Evaluation page is the primary dissertation evidence page. It compares System Uptime, Recorded Incidents, Fault Detection Time, Response Time, and Outage Duration before and after InfraSight. It highlights 81 percent reduction in fault detection time, 65 percent reduction in response time, 68 percent reduction in outage duration, and 5.4 percent improvement in uptime. The page also shows model performance cards, confusion matrix, feature importance chart, before vs after chart, system activity summary, JSON export, and PDF export.

Section 18: User Guide - Settings

The Settings page displays threshold settings, polling settings, model settings, and system information. Thresholds include CPU, memory, packet loss, latency, interface error, and uptime critical values. Polling interval defaults to 30 seconds. Model settings show version, training date, accuracy, and samples. System information includes Firebase project ID infrasight-gzu, backend version, frontend version, and environment mode.

Section 19: Developer Guide - Local Setup

Prerequisites include Python 3.11 or higher, Node.js 18 or higher, Git, and a configured Firebase project. Repository structure contains infrasight-api, infrasight-web, docs, verify-all.ps1, .gitignore, and README.md. Backend setup uses python -m venv .venv, activation, and pip install -r requirements.txt. Frontend setup uses npm install. Verification uses .\verify-all.ps1 and should end with ALL CHECKS PASSED.

Section 21: Deployment Guide

GitHub setup uses git remote add origin, git branch -M main, and git push -u origin main. Backend deployment on Render uses root directory infrasight-api, build command pip install -r requirements.txt, and start command uvicorn app.main:app --host 0.0.0.0 --port $PORT. Required Render variables include FIREBASE_PROJECT_ID, FIREBASE_CLIENT_EMAIL, FIREBASE_PRIVATE_KEY, FIREBASE_PRIVATE_KEY_ID, FIREBASE_CLIENT_ID, and MOCK_MODE=false. Frontend deployment on Vercel uses root directory infrasight-web, build command npm run build, output directory dist, and VITE environment variables including VITE_API_BASE_URL. Firebase Authentication authorized domains must include the Vercel domain.

Section 22: Security Considerations

Never commit infrasight-api/.env, infrasight-web/.env, infrasight-web/.env.production, Firebase service account JSON, infrasight-api/.venv, infrasight-web/node_modules, infrasight-web/dist, or infrasight-api/app/ml/models. Firebase credentials must be stored as environment variables. Firestore rules should enforce authentication. API CORS should allow trusted origins only.

Section 23: Troubleshooting Guide

If the backend will not start, check virtual environment activation, requirements installation, .env values, and run .\.venv\Scripts\python.exe -c "import app.main". If the frontend will not build, run npm install and check import paths. If Firebase shows mock mode, check FIREBASE_SERVICE_ACCOUNT_PATH and MOCK_MODE=false. If login fails, check Firebase Authentication Email/Password provider and web config. If predictions are wrong, check model file and feature order. If verify-all fails, fix the first FAIL message and rerun. If charts do not render, confirm recharts is installed and chart data arrays are not empty.

Section 26: Conclusion

InfraSight is a complete, verified, and deployable ICT infrastructure monitoring and predictive maintenance system designed for Great Zimbabwe University. The system demonstrates how machine learning can improve fault detection, reduce response time, and support proactive maintenance decisions. Key achievements include a Random Forest prediction engine using eight telemetry variables, 81 percent reduction in fault detection time, live Firebase cloud integration, complete alert management, four validated scenarios, production-ready Vercel and Render configuration, examiner-ready Demo Mode, and PDF evidence export. InfraSight provides a strong academic contribution by combining real-time monitoring, machine learning, cloud services, and evidence-based evaluation into one demonstrable dissertation prototype.
